# Localize User-Facing Skill Prompts Implementation Plan

> **For agentic workers:** Use team-driven-development to execute this plan.

**Goal:** Insert a canonical `## Language Policy` block into every team-dd `SKILL.md` so the skills render user-facing strings (announce, gates, status, errors) in the user's conversation language, and document the source-vs-runtime distinction in `guidelines/writing.md`.
**Architecture:** Two sequential documentation tasks. Task 1 inserts the identical Language Policy block into six `SKILL.md` files at the documented anchor (after Announce/HARD-GATE, before the first procedural section). Task 2 appends a "Source vs. runtime" bullet to the `## Language` section of `guidelines/writing.md`. Each task uses exact `old_string → new_string` Edit calls, verifies with `grep`, and commits.
**Tech Stack:** Markdown.
**Spec:** docs/team-dd/specs/2026-04-18-localize-user-facing-prompts-design.md (authoritative; consult for rationale/decisions)
**Sprints:** sprints/2026-04-18-localize-user-facing-prompts/ (Sprint Contract files, generated by sprint-master)

The canonical Language Policy block (identical in all six files):

````markdown
## Language Policy

Render fixed user-facing text in the user's conversation language: announce lines, gate prompts (spec / plan / execution / cleanup), surfaced concerns, status and progress reports, and user-facing error or usage messages. If the user explicitly requests another language for the session, use that instead. Treat the English phrasings in this file as semantic templates — translate faithfully, preserve placeholder variables (`<path>`, `<N>`, ...), and keep Markdown structure.

Keep literal regardless of language: shell and slash commands, file paths, machine-parsable identifiers (`PASS`, `APPROVE`, `DONE`, `BLOCKED`, `CHANGES_REQUESTED`, severity / disposition labels, `MET` / `NOT_MET`), status-line markers (📌 🔍 ❓ ⚠), Markdown headings used as section anchors (e.g., `## Completion Report`), and column headers of report tables.

Detection: default to the dominant language of the user's recent natural-language messages. If the user's input is pure code, paths, or commands (no natural language), keep the previous turn's language; use English on the very first turn if no prior signal exists. Override persists until the user changes it again.
````
